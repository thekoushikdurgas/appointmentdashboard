## 1. Architecture (doc + API)

- **Gateway** talks to **s3storage** (Go) via **`S3StorageEC2Client`** (`S3STORAGE_SERVER_API_URL`, `X-API-Key`; legacy `LAMBDA_S3STORAGE_*` still accepted by the API).
- **Logical bucket:** `user.bucket` or `user.uuid`; keys are **CSV-oriented** for list/preview (doc: CSV-only listing).
- **Multipart CSV path (doc):**  
  `s3.initiateCsvUpload` → **`upload.presignedUrl` / `upload.registerPart`** per part → **`s3.completeCsvUpload`** with **`CompleteUploadInput`** (same as **`upload.completeUpload`**).

**Queries (implemented names in `s3/queries.py`):**

| Field               | Args                         | Returns                                                                                                                                                                            |
| ------------------- | ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `s3Files`           | `prefix`                     | `S3FileList` (`files`, `total`, `bucketDisplayName`)                                                                                                                               |
| `s3FileData`        | `fileKey`, `limit`, `offset` | `S3FileData` (`fileKey`, `rows[]` with **`data: JSON`**, `limit`, `offset`, `totalRows`)                                                                                           |
| `s3FileInfo`        | `fileKey`                    | **`S3FileInfo`** (`key`, `filename`, `size`, `lastModified`, `contentType`)                                                                                                        |
| `s3FileDownloadUrl` | `fileKey`, `expiresIn`       | `S3DownloadUrlResponse`                                                                                                                                                            |
| `s3FileSchema`      | `fileKey`                    | `[FileSchemaColumn!]!`                                                                                                                                                             |
| `s3FileStats`       | `fileKey`                    | **`S3FileStats`** — API has **`rowCount`** + **`columns` (JSON map)**, not the richer doc snippet with `totalRows` / `totalColumns` / `fileSize` unless doc was updated separately |
| `s3BucketMetadata`  | —                            | **`JSON!`** (arbitrary object from s3storage), not a fixed GraphQL object                                                                                                          |

**Mutations (`s3/mutations.py`):**

| Mutation            | Input                                                         | Notes                                                                                             |
| ------------------- | ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `initiateCsvUpload` | `InitiateCsvUploadInput` (`filename`, **`fileSize: BigInt`**) | `.csv` required; delegates to **`upload.initiateUpload`** with `content_type: text/csv`           |
| `completeCsvUpload` | **`CompleteUploadInput`**                                     | **Only `uploadId`** in `upload/inputs.py`; parts are already stored via **`upload.registerPart`** |
| `deleteFile`        | `fileKey: String!`                                            | Boolean                                                                                           |

**`CompleteUploadResponse` (upload types):** `status`, `fileKey`, `s3Url`, `location` — **not** the `S3FileInfo`-style shape the app uses for “complete CSV”.

## 2. App layer: what matches vs what does not

**Aligned (or close):**

- **`listFiles`** → `s3Files` + fields match doc/UI needs.
- **`getDownloadUrl`** → `s3FileDownloadUrl`.
- **`initiateCsvUpload`** → correct operation name; **`fileSize`** should be sent in a form GraphQL **`BigInt`** accepts (often **string** for large values is OK with your codegen — verify).
- **`deleteFile`** → matches.
- **`useS3Files`** list/delete/download path used by **`files/page.tsx`** matches the working subset above.

**Misaligned (will not match the live schema):**

1. **Wrong query field names** in `s3Service.ts`: the API exposes **`s3FileData`**, **`s3FileInfo`**, **`s3FileSchema`**, **`s3FileStats`**, **`s3BucketMetadata`**, but the service uses **`fileData`**, **`fileInfo`**, **`fileSchema`**, **`fileStats`**, **`bucketMetadata`**.

2. **`s3FileData` response shape:** Rows are **`{ data: JSON }`**, not a flat row object. The service types **`S3FileDataResult.rows`** as `S3FileDataRow[]` keyed records — needs **unwrapping `row.data`**.

3. **`s3FileInfo`:** API returns **`S3FileInfo`** (`key`, `filename`, `size`, …). The app expects **`S3FileInfoRow`** (`fileKey`, `fileName`, `uploadedAt`, `status`, …) — **different contract**.

4. **`completeCsvUpload`:** Uses mutation input type **`CompleteCsvUploadInput`** and sends **`uploadId`, `fileKey`, `s3UploadId`, `parts`**. The schema expects **`CompleteUploadInput`** with **`uploadId` only** (parts are registered separately). **`useS3Files.uploadCsv`** is built around the invalid payload.

5. **`uploadService.ts`** **`completeUpload`** — if it still sends **`parts`** in the mutation variables, that also disagrees with **`CompleteUploadInput`** (upload_id only). Worth verifying in Agent mode.

6. **`s3BucketMetadata`:** Treated as a typed **`S3BucketMetadata`** object; API returns **`JSON`**.

7. **`s3FileStats`:** App expects **`totalRows`, `totalColumns`, `fileSize`, `columns[]`**; API returns **`rowCount` + `columns` JSON** — adjust types and UI.

## 5. Summary

- **`07_S3_MODULE.md`** and **`contact360.io/api/app/graphql/modules/s3/`** describe a coherent **S3 + multipart CSV + upload module** design.
- **`contact360.io/app`** **`s3Service` / `useS3Files`** are **only partially correct**: **list, download URL, initiate, delete** are in the right direction, but **secondary queries use non-existent field names**, **row/preview and stats shapes don’t match the API**, **`completeCsvUpload` variables are invalid**, and **bucket metadata is typed as a fixed object though the API returns JSON**.
- **Files page** works for **list / download / delete** paths that hit the correct operations; **full preview/stats/upload completion** need the fixes above before they match **`07_S3_MODULE.md`** end-to-end.
